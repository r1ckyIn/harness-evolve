    **Step 1: Create src/analysis/classifiers/code-corrections.ts**

    Export `classifyCodeCorrections(summary, snapshot, config): Recommendation[]`.

    Heuristic for v1 (per RESEARCH.md Open Question 2):
    - The pre-processor summary does NOT track per-tool failure rates directly
    - `summary.tool_frequency` includes all tool events (pre, post, failure) counted together
    - v1 approach: Look at tool_frequency entries. For tools with very high counts (>= 20 uses), suggest reviewing for potential rule creation. This is a LOW-confidence heuristic.
    - Iterate `summary.tool_frequency`. For each tool where `count >= config.thresholds.code_correction_min_failures` (default 3) — produce a LOW confidence RULE recommendation only if the tool count is notably high (top quartile of all tool counts, or count >= 20 as a simple heuristic)
    - Actually simplified further: for each entry in tool_frequency, if the tool_name suggests a code-modification tool (Write, Edit, MultiEdit — tools that modify files) and count is high (>= 20), suggest a RULE for that file type pattern
    - Create Recommendation with:
      - `id`: `rec-correction-${index}`
      - `target`: `'RULE'`
      - `confidence`: `'LOW'`
      - `pattern_type`: `'code_correction'`
      - `title`: `Frequent code modifications with ${entry.tool_name} (${entry.count} uses)`
      - `description`: `The ${entry.tool_name} tool has been used ${entry.count} times. Review for recurring patterns that could become a coding rule or convention.`
      - `evidence`: `{ count: entry.count, examples: [entry.tool_name] }`
      - `suggested_action`: `Review recent ${entry.tool_name} usage for recurring patterns. If a consistent code style or approach emerges, create a rule in .claude/rules/.`

    - Create Recommendation with:
